# MERGE ARENA

A dependency-free static Telegram Mini App game (vanilla HTML/CSS/JS). All state is persisted client-side in `localStorage`; there is no backend or database, and no build step for the game itself. Files: `index.html`, `arena.js`, `arena.css`, `manifest.json`, `service-worker.js`, `icons/`. Deployed to GitHub Pages via `.github/workflows/pages.yml`.

## Cursor Cloud specific instructions

- There **is** a `package.json` now, but only for a `jsdom`-based regression test suite — the game itself still has no build step. Run `npm install` once, then `npm test` (or `node tests/arena.test.js`) before pushing any change. CI runs this on every push/PR (`.github/workflows/test.yml`) and again before every deploy (`.github/workflows/pages.yml`); a failing suite blocks the live site from updating.
- **Run it in a browser** by serving the repo root over HTTP (needed so `localStorage`, the service worker, and the Telegram SDK behave correctly): `python3 -m http.server 8000` from `/workspace`, then open `http://localhost:8000/index.html`. Any static server works.
- The game is **fully playable in a plain browser**. The Telegram WebApp SDK and Google Fonts load from CDNs and degrade gracefully; Telegram-only features (haptics, header color) are optional and only testable inside a Telegram client.
- **The Shop checkout is a free demo, not a real payment.** `confirmPay()` in `arena.js` just shows a `window.confirm` dialog and grants the item — no Telegram Stars are ever charged. This is intentionally, visibly disclosed in the UI (a banner on the Shop tab, a "FREE — DEMO" badge on every offer card, and a banner in the pay-confirmation modal). Do not remove that disclosure or make the flow look like a real charge without actually wiring up real Telegram Stars payments (which requires a bot backend — see below).
- **Real Telegram Stars payments require a backend that does not exist here.** To make Shop purchases real: (1) register a bot via BotFather, (2) build a server that calls the Bot API's `createInvoiceLink` and opens it client-side via `Telegram.WebApp.openInvoiceLink()`, and (3) grant the reward only after the bot receives a `successful_payment` webhook — never trust a client-side "payment succeeded" callback alone. This is a real engineering project, not a config change.
- **The "Invite Friends" share button intentionally grants no reward.** There is no backend to verify who was actually referred, so any reward-for-share mechanic would be a free, unlimited currency farm (tap Share, cancel, repeat). If a real referral program is wanted later, it needs server-side attribution tied to Telegram user IDs.
- All game state (gems, trophies, energy, achievements, sound preference) lives in the `merge_arena_v2` `localStorage` key with no server validation, so any user can edit it via browser devtools. This is inherent to the current client-only architecture and is not something to "fix" without adding a backend — treat it as a known, accepted limitation, not a bug to patch with client-side obfuscation.
- Core loop to smoke-test: tap **Get Hero** (costs 1 ⚡), drag two identical heroes together to merge/upgrade them, then tap **Fight!**. Every 5th level is a boss (2x rewards). Progress persists across reloads via the `merge_arena_v2` localStorage key; clear that key to reset game state.
